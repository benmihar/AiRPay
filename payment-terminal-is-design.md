# Проект учебной информационной системы

## Тема
Веб-информационная система, эмулирующая работу платежного терминала самообслуживания.

## 1. Назначение системы
Система предназначена для учебной демонстрации принципов работы платежного терминала в формате веб-сайта. Пользователь взаимодействует с интерфейсом, похожим на экран терминала: выбирает услугу, вводит реквизиты, указывает сумму, вносит виртуальную оплату и получает электронный чек.

Система не выполняет реальные платежи. Все операции являются имитационными и используются для обучения, тестирования интерфейсов и демонстрации бизнес-процессов.

## 2. Цель разработки
Разработать учебную ИС, которая:
- имитирует основные функции платежного терминала;
- позволяет изучить пользовательский сценарий оплаты услуг;
- демонстрирует обработку транзакций, валидацию данных и формирование чеков;
- предоставляет преподавателю или администратору средства просмотра истории операций.

## 3. Объект автоматизации
Процесс приема платежей через терминал самообслуживания за типовые услуги:
- мобильная связь;
- интернет;
- ЖКХ;
- штрафы;
- переводы на виртуальный счет;
- пополнение электронного кошелька.

## 4. Предметная область
Платежный терминал представляет собой устройство самообслуживания, через которое пользователь может оплачивать услуги без участия оператора. В учебной веб-версии воспроизводятся ключевые этапы:
- выбор категории услуги;
- выбор поставщика;
- ввод идентификатора плательщика;
- ввод суммы;
- проверка корректности данных;
- имитация внесения денежных средств;
- завершение операции;
- печать или просмотр чека.

## 5. Пользователи системы

### 5.1. Клиент
Основной пользователь, который:
- выбирает услугу;
- вводит платежные данные;
- подтверждает операцию;
- получает чек.

### 5.2. Администратор
Пользователь с расширенными правами, который:
- управляет списком услуг и поставщиков;
- просматривает журнал операций;
- отслеживает статистику платежей;
- изменяет параметры эмуляции.

### 5.3. Преподаватель
Может использовать систему для демонстрации сценариев работы терминала, анализа ошибок и проверки результатов выполнения лабораторной работы.

## 6. Функциональные требования

### 6.1. Функции клиентской части
Система должна обеспечивать:
- отображение стартового экрана терминала;
- выбор категории платежа;
- выбор поставщика услуг;
- ввод номера телефона, лицевого счета, номера договора или иного идентификатора;
- ввод суммы платежа;
- автоматический расчет комиссии;
- отображение итоговой суммы к оплате;
- имитацию внесения купюр или оплату виртуальным способом;
- подтверждение транзакции;
- формирование электронного чека;
- возврат на главный экран по тайм-ауту.

### 6.2. Функции административной части
Система должна обеспечивать:
- авторизацию администратора;
- просмотр списка транзакций;
- фильтрацию транзакций по дате, услуге, статусу;
- просмотр деталей операции;
- ведение справочника услуг;
- ведение справочника поставщиков;
- настройку комиссии;
- просмотр сводной статистики.

### 6.3. Учебные функции
Система должна обеспечивать:
- запуск успешного и ошибочного сценария оплаты;
- генерацию типовых ошибок ввода;
- имитацию недостаточной суммы;
- демонстрацию отмены операции;
- хранение истории учебных транзакций.

## 7. Нефункциональные требования
- Веб-интерфейс должен корректно работать в современных браузерах.
- Интерфейс должен быть адаптирован под сенсорное взаимодействие.
- Основные действия должны выполняться не более чем за 2 секунды.
- Система должна быть безопасной на учебном уровне: пароли хранятся в хешированном виде, доступ в админ-панель ограничен.
- Интерфейс должен быть интуитивно понятным и максимально приближен к реальному терминалу.
- Все реальные платежные шлюзы должны быть отключены; используются только тестовые данные.

## 8. Основные бизнес-процессы

### 8.1. Сценарий успешной оплаты
1. Пользователь открывает главный экран.
2. Выбирает категорию услуги.
3. Выбирает поставщика.
4. Вводит идентификатор платежа.
5. Вводит сумму.
6. Система рассчитывает комиссию и итог.
7. Пользователь подтверждает оплату.
8. Система имитирует прием денежных средств.
9. Создается транзакция со статусом `успешно`.
10. Пользователю отображается чек.

### 8.2. Сценарий ошибочного ввода
1. Пользователь выбирает услугу.
2. Вводит некорректный номер счета или телефона.
3. Система выполняет проверку формата.
4. Отображается сообщение об ошибке.
5. Пользователь исправляет данные или отменяет операцию.

### 8.3. Сценарий отмены операции
1. Пользователь проходит часть шагов.
2. Нажимает кнопку отмены.
3. Система удаляет временные данные сессии.
4. Происходит возврат на главный экран.

## 10. Логическая архитектура системы

### 10.1. Архитектурный стиль
Для учебного проекта подходит трехуровневая архитектура:
- уровень представления: веб-интерфейс;
- уровень бизнес-логики: обработка сценариев оплаты;
- уровень данных: база данных и журналы транзакций.

### 10.2. Компоненты системы

#### Frontend
- интерфейс терминала;
- формы ввода;
- экранные клавиатуры;
- таймер бездействия;
- отображение статусов операции;
- модуль генерации чека.

#### Backend
- модуль авторизации администратора;
- модуль управления услугами и поставщиками;
- модуль обработки транзакций;
- модуль расчета комиссии;
- модуль валидации данных;
- модуль отчетности.

#### Database
- хранение пользователей;
- хранение справочников услуг;
- хранение поставщиков;
- хранение транзакций;
- хранение настроек комиссии;
- хранение логов действий.

## 11. Предлагаемый стек технологий

### Вариант для учебной реализации
- Frontend: HTML, CSS, JavaScript или React
- Backend: Node.js + Express или Python + Django/Flask
- База данных: PostgreSQL или MySQL
- Аутентификация: сессии или JWT для админ-панели

### Рекомендуемый минимальный стек
- Frontend: HTML, CSS, JavaScript
- Backend: Node.js + Express
- DB: PostgreSQL

Такой стек прост для демонстрации, хорошо документирован и подходит для курсового проекта.

## 13. Пример API

### Клиентские маршруты
- `GET /api/categories` - получить список категорий
- `GET /api/providers?categoryId=1` - получить поставщиков
- `POST /api/validate-account` - проверить введенные реквизиты
- `POST /api/calculate` - рассчитать комиссию и итог
- `POST /api/transactions` - создать учебную транзакцию
- `GET /api/receipts/{id}` - получить чек

### Административные маршруты
- `POST /api/admin/login` - вход
- `GET /api/admin/transactions` - список транзакций
- `GET /api/admin/stats` - статистика
- `POST /api/admin/providers` - добавить поставщика
- `PUT /api/admin/providers/{id}` - изменить поставщика
- `DELETE /api/admin/providers/{id}` - деактивировать поставщика

## 14. Интерфейсные требования

### Пользовательский интерфейс терминала
- крупные кнопки;
- минимальное количество текста;
- контрастные цвета;
- пошаговая навигация;
- экранная цифровая клавиатура;
- индикатор текущего шага;
- кнопки `Назад`, `Отмена`, `Подтвердить`.

### Интерфейс администратора
- таблицы с фильтрами;
- карточки статистики;
- формы редактирования справочников;
- отдельный экран просмотра чеков и операций.

## 15. Требования к безопасности
- реальная обработка денег отсутствует;
- используются тестовые счета и фиктивные номера;
- пароли администраторов не хранятся в открытом виде;
- должна быть защита от прямого доступа к административным маршрутам;
- должен вестись журнал действий администратора;
- необходимо ограничить количество попыток входа.

## 16. Возможные статусы транзакции
- `new` - операция создана;
- `validated` - данные проверены;
- `awaiting_payment` - ожидание внесения средств;
- `paid` - оплата успешно завершена;
- `failed` - ошибка операции;
- `cancelled` - операция отменена.

## 17. Отчеты и аналитика
Администратор должен иметь возможность:
- просматривать количество операций за период;
- видеть общую сумму учебных платежей;
- анализировать популярные категории услуг;
- отслеживать долю успешных и отмененных операций;
- выгружать журнал транзакций в CSV.
